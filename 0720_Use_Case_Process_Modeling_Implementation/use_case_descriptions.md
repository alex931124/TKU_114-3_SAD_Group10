# 使用案例描述（Use Case Descriptions）

## UC-01 提交報修

- 目標：宿舍學生可提交一筆完整報修申請並確認案件已受理。
- 主要參與者：宿舍學生
- 支援參與者：無
- 觸發事件：學生於報修頁面點選送出。
- 前置條件：宿舍學生已進入報修表單頁面。
- 成功後置條件：建立新案件，系統顯示案件編號與已提交狀態。
- 最低保證：缺漏欄位時不建立不完整案件，並顯示錯誤訊息。
- 主要流程：
  1. 學生選擇宿舍、填寫房號、設備類型與問題描述。
  2. 系統檢查必填欄位是否完整。
  3. 系統建立新案件並回傳案件編號。
  4. 系統顯示「已提交」狀態。
- 替代流程：A1：資料缺漏時，系統保留已填資料並請學生補正。
- 例外流程：E1：若房號或問題描述未填，系統拒絕送出並停留在表單頁面。
- 輸入資料：宿舍、房號、設備類型、問題描述
- 輸出資料：案件編號、報修狀態
- 狀態變化：空白 / 未送出 → 已提交
- 對應需求：FR-01、FR-02、NFR-01
- 對應驗收條件：AC-01-01、AC-01-02、AC-01-03

## UC-03 派工給維修人員

- 目標：宿舍管理員將待處理案件指定給維修人員。
- 主要參與者：宿舍管理員
- 支援參與者：維修人員
- 觸發事件：管理員選擇維修人員並按派工。
- 前置條件：案件已存在於待處理清單中。
- 成功後置條件：案件狀態更新為已派工，並保留指派人員。
- 最低保證：未選擇人員時不得建立錯誤派工紀錄。
- 主要流程：
  1. 管理員查看待處理清單。
  2. 管理員選擇指定維修人員。
  3. 系統更新案件狀態為已派工。
  4. 系統顯示指定維修人員名稱。
- 替代流程：A1：未指派維修人員，系統提示需先選擇。
- 例外流程：E1：若案件已不存在或狀態異常，系統不執行派工。
- 輸入資料：案件編號、指定維修人員
- 輸出資料：已派工狀態、維修人員欄位
- 狀態變化：待派工 → 已派工
- 對應需求：FR-03、FR-04
- 對應驗收條件：AC-02-01、AC-02-02

## UC-05 完成維修回報

- 目標：維修人員可完成已派工任務並回傳完成狀態。
- 主要參與者：維修人員
- 支援參與者：宿舍管理員
- 觸發事件：維修人員於已派工清單點擊標記完成。
- 前置條件：案件已狀態為已派工。
- 成功後置條件：案件狀態更新為已完成。
- 最低保證：不得重複建立多筆完成紀錄。
- 主要流程：
  1. 維修人員查看已派工任務清單。
  2. 維修人員點擊「標記完成」。
  3. 系統更新狀態為已完成。
  4. 系統回到清單頁面顯示完成狀態。
- 替代流程：A1：如果案件已完成，系統不重複執行。
- 例外流程：E1：如果案件狀態異常，系統維持原狀態並提示錯誤。
- 輸入資料：案件編號、維修人員操作
- 輸出資料：已完成狀態
- 狀態變化：已派工 → 已完成
- 對應需求：FR-05
- 對應驗收條件：AC-03-01、AC-03-02
